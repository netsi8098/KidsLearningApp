# Staging Fidelity Audit

Last updated: 2026-03-26

This document tracks parity between staging and production environments, identifies known gaps, and provides the release promotion checklist.

## Staging vs Production Comparison

| Dimension | Production | Staging | Gap | Fix |
|-----------|-----------|---------|-----|-----|
| Database | RDS Multi-AZ, r6g.large | RDS Single-AZ, t3.medium | Failover not tested | Accept for cost |
| Redis | ElastiCache cluster, 3 nodes | ElastiCache single node | No cluster mode | Accept for cost |

## Staging Safeguards

### Data Protection
- No real user data is ever copied to staging
- All staging data is synthetic, generated by `seed-staging.ts`
- Staging uses a separate S3 bucket with a 30-day lifecycle policy (auto-delete)
- Database is reset weekly via scheduled CI job

### External Service Isolation
- **Email**: Sending is routed to Mailtrap; no real emails are delivered
- **Payments**: Stripe test mode only; use test card `4242 4242 4242 4242`
- **AI APIs**: Separate test accounts with low quotas to prevent cost overruns
- **Push notifications**: Disabled in staging; logged to console instead
- **Analytics**: Separate staging project in analytics provider; no prod contamination

### Access Control
- Staging is behind VPN or IP allowlist
- Staging admin passwords are different from production (all use `staging123`)
- Staging JWT secret is different from production
- No production service accounts have access to staging resources

### Cost Controls
- Staging auto-scales are capped (max 2 API, max 1 worker)
- S3 storage has 30-day lifecycle deletion
- Redis has no persistence (volatile)
- RDS is single-AZ, t3.medium (no Multi-AZ overhead)
- CloudFront uses a single price class (PriceClass_100)

### Running Seeds

```bash
# Standard staging seed
cd backend && npm run db:seed:staging

# Full staging + edge cases
./scripts/seed-staging.sh --edge-cases

# Reset and re-seed from scratch
./scripts/seed-staging.sh --reset --edge-cases -y
```

## Release Fidelity Checklist

Before promoting staging to production, complete every item below.

### Pre-Release (Staging Validation)

- [ ] All staging CI/CD tests pass (unit, integration, contract, smoke)
- [ ] Staging health checks green for 1+ hour after deploy
- [ ] No new error groups with count > 10 in staging
- [ ] Manual smoke test of critical paths:
  - [ ] Admin login and content CRUD
  - [ ] Parent signup and onboarding flow
  - [ ] Child profile creation and content playback
  - [ ] Subscription upgrade flow (Stripe test mode)
  - [ ] Offline mode: disconnect network, verify cached content loads
  - [ ] Bedtime mode toggle and content filtering
  - [ ] PWA install prompt (mobile Chrome)
- [ ] Feature flags match production defaults (run `flag-sync` script)
- [ ] Database migration ran successfully in staging without data loss
- [ ] No error rate increase during 1-hour staging soak period
- [ ] Bundle size within budget (main chunk < 500KB gzip)
- [ ] BullMQ queue backlog is zero
- [ ] Performance baselines not regressed (check p95 metrics)

### Release Execution

- [ ] Create release tag in git
- [ ] Run production migration (with rollback plan documented)
- [ ] Deploy backend first, then frontend
- [ ] Verify CloudFront invalidation completed
- [ ] Confirm service worker update propagates (check `/sw.js` version)
- [ ] Monitor error rates for 30 minutes post-deploy
- [ ] Verify key API endpoints return 200
- [ ] Spot-check 3 content items render correctly

### Post-Release

- [ ] Confirm zero increase in 5xx error rate
- [ ] Check queue processing latency is normal
- [ ] Verify new feature flags are in expected state
- [ ] Update release notes / changelog
- [ ] Notify team in Slack channel
- [ ] Schedule 24-hour check-in for delayed issues

## Known Staging Limitations

1. **No real push notifications**: Notifications are logged to console. Test notification rendering via the admin preview panel instead.

2. **Stripe test mode differences**: Some webhook events (e.g., `invoice.payment_failed`) behave differently in test mode. Always verify payment flows in Stripe dashboard.

3. **Single-region CDN**: Staging CloudFront only uses us-east-1 edge locations. Global latency testing requires synthetic monitoring tools.

4. **No load testing by default**: Staging infrastructure is undersized. For load tests, temporarily scale up RDS and API instances, then scale back down.

5. **Redis volatility**: Staging Redis has no persistence. BullMQ jobs are lost on Redis restart. This is acceptable because staging queues process synthetic data.

6. **AI API rate limits**: Staging uses low-quota test API keys. Brief generation and voice pipeline may hit rate limits faster than production.
